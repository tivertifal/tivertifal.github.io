How to make sure leaderboards can be trusted

I've been thinking about how to have leaderboards for a roguelike like the one I'm planning. My first thought in this regard is that leaderboards only make sense if the data they're based on is credible. I've played many games where the top leaderboard entries had clearly been manipulated in some way. When that happens, even if it happens only once, the point of the leaderboards evaporates instantly, as does the appeal of competing for a spot on them. I've looked at different ways to make sure data being transmitted to the leaderboards (let's call these "the server" and contrast it with "the client" from now on) could be checked and verified and rejected if it was manipulated. Unsurprisingly, it turns out there are quite a lot of attack vectors for any sort of setup where a client sends data to a server.

I'll be making a game that's played offline with the ability to save and resume. Potentially, a player might play for a while, save the game, take an extended break, and come back and finish the run three years later. This puts certain constraints on how I can verify the integrity of a submitted run, because a bad actor would theoretically have infinite time to circumvent my safeguards.

While trying to solve this problem, one of the things I looked at was how realtime strategy games like Starcraft work. They use something called deterministic lockstep, which is quite ingenious. It's intended for multiple clients connected to a single server playing against each other in realtime, but it turns out I can use a modified version of this design pattern for my offline, glacial-pace roguelike.

Determinism

There are two ways to think about determinism when it comes to games. There's a sort of front-end, user-facing determinism, which I think is what most people think of when they hear the word. This kind of determinism has to do with how random the effect of actions is from the point of view of the user. When you fire a missile weapon, does it deal exactly 15 damage, or does it deal a random amount of damage between 10 and 30? When you step away from an enemy, do they always pursue in a straight line, or do they randomly choose between a number of different actions? In this sense of the word, a highly deterministic game is one where you can predict the outcome of your actions with a high degree of certainty, and those games are akin to puzzles. There's a way to handle a given challenge that results in the best possible outcome, and if you know enough about the mechanics involved, you can predict this path with great certainty. I think this kind of determinism is no better or worse than the alternative as a way to build a game; it tests different skills, poses different kinds of challenges, and requires different kinds of strategy than less deterministic games.

In the context of discussing data verification and leaderboards, however, it's a different kind of determinism I'm interested in. Using the same kind of terminology as before, we might call this back-end, hidden determinism. This kind of determinism has nothing to do with the user's experience, but describes what happens under the hood whenever the player takes a given action in a given context. That description is kind of vague in itself, so let me make it more concrete by circling back to the deterministic lockstep I mentioned earlier and explain how back-end determinism relates to the leaderboards I want to implement in the game.
