How to make sure leaderboards can be trusted

I've been thinking about how to have leaderboards for a roguelike like the one I'm planning. My first thought in this regard is that leaderboards only make sense if the data they're based on is credible. I've played many games where the top leaderboard entries had clearly been manipulated in some way. When that happens, even if it happens only once, the point of the leaderboards evaporates instantly, as does the appeal of competing for a spot on them. I've looked at different ways to make sure data being transmitted to the leaderboards (let's call these "the server" and contrast it with "the client" from now on) could be checked and verified and rejected if it was manipulated. Unsurprisingly, it turns out there are quite a lot of attack vectors for any sort of setup where a client sends data to a server.

I'll be making a game that's played offline with the ability to save and resume. Potentially, a player might play for a while, save the game, take an extended break, and come back and finish the run three years later. This puts certain constraints on how I can verify the integrity of a submitted run, because a bad actor would theoretically have infinite time to circumvent my safeguards.

While trying to solve this problem, one of the things I looked at was how realtime strategy games like Starcraft work. They use something called deterministic lockstep, which is quite ingenious. It's intended for multiple clients connected to a single server playing against each other in realtime, but it turns out I can use a modified version of this design pattern for my offline, glacial-pace roguelike.

Determinism

There are two ways to think about determinism when it comes to games. There's a sort of front-end, user-facing determinism, which I think is what most people think of when they hear the word. This kind of determinism has to do with how random the effect of actions is from the point of view of the user. When you fire a missile weapon, does it deal exactly 15 damage, or does it deal a random amount of damage between 10 and 30? When you step away from an enemy, do they always pursue in a straight line, or do they randomly choose between a number of different actions? In this sense of the word, a highly deterministic game is one where you can predict the outcome of your actions with a high degree of certainty, and those games are akin to puzzles. There's a path to solving a given challenge that results in the best possible outcome, and if you know enough about the mechanics involved, you can predict this path with great certainty. I think this kind of determinism is no better or worse than the alternative as a way to build a game; it tests different skills, poses different kinds of challenges, and requires different kinds of strategy than less deterministic games.

In the context of discussing data verification and leaderboards, however, it's a different kind of determinism I'm interested in. Using the same kind of terminology as before, we might call this back-end, hidden determinism. This kind of determinism has nothing to do with the user's experience, but describes what happens under the hood whenever the player takes a given action in a given context. That description is kind of vague in itself, so let me make it more concrete by circling back to the deterministic lockstep I mentioned earlier and explain how back-end determinism relates to the leaderboards I want to implement in the game.

Deterministic lockstep and the command pattern

The command pattern is a design pattern from object-oriented programming. The basic idea is to treat a command as a discrete object, separate from the invoker and receiver of the command. It's kinda like chess notation, where you can write down an entire game from start to finish as discrete commands: 1. e4 e5 2. Nf3 Nc6 3. Bb5 a6 and so on. It doesn't matter who played white and who played black, or whether the game was played online or in real life, or in 1950 or 2026; the game is the same in all cases when expressed as pure commands. The pattern has a number of advantages. First of all, it gives us the ability to undo and redo commands, to rewind and fast forward, to pause, analyze, evaluate states at any given time, and so on. More importantly, at least for my purposes, it lets us recreate the entire chain of commands from start to finish at any point in time with the exact same outcome.

Lockstep is what you see at military parades, when the soldiers march in perfect synchronization, each person taking the exact same step at the exact same time as all the other people. Deterministic lockstep borrows this imagery because it's a way to make sure all the players' local clients are similarly in sync. It works by implementing the command pattern so players exchange input commands instead of game states. Each player's game bundles up the local player's inputs for a given time period and transmits these to the server. The server then bundles and redistributes all the commands to all the players, and each player's game drives the local simulation forward using the commands it received from the server. This ensures that all the local simulations are seeing the exact same outcomes; since the only thing being exchanged is intent, and this intent is validated by the server as permissible intent in a given context, it ensures the integrity of the game and protects against cheating.

One particular thing about the simulation that each player's client is running is required for the above description to be true: the simulation must be entirely deterministic. If a game is played on eight different machines running eight different simulations, using only the players' intent and commands as input, then each of those eight simulations must arrive at the exact same outcome of each action, otherwise the whole thing falls apart. To explain why that's interesting in the context of creating a traditional roguelike with leaderboards, let me try to tie the two things together in a meaningful way.

Verification by replay

Reading about deterministic lockstep made me think I'd found a solution to the problem of verifiable leaderboard submissions. If I could reduce an entire playthrough of the game to a series of inputs, and I was running the exact same simulation on the server as the simulation running on the client, then I could just feed the submitted input into the server's version of the simulation and repeat the whole playthrough. Assuming each input was legal and valid for a given state of the game, and I arrived at the same outcome at the end of the sequence of inputs, then the playthrough would be valid and its integrity verified. It wouldn't matter if the player had three years to manipulate the data, because if the data consisted entirely of valid commands, then manipulation of that data would be pointless.

So, I know conceptually how to make sure my leaderboards will be credible. I need a completely deterministic game engine that lets me replay each submitted playthrough from beginning to end, and then compare end states. If I was making a client for playing chess, that would be easy enough to implement, but I'm making a game in a genre that is defined by randomness. But while it may seem like the type of game and the methods I've chosen to solve the leaderboard issue are fundamentally incompatible, luckily that isn't the case.
